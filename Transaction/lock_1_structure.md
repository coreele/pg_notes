# Lock structure

## 重量级锁表

### 1. 定位与性质

* **存储位置**：完全驻留在 **共享内存（Shared Memory）** 中，所有 Backend 进程可见。
* **管理对象**：表（Relation）、事务（Transaction ID）、页（Page）等逻辑对象。
* **核心特性**：支持任意粒度的锁模式冲突检测、**死锁检测**，以及跨进程的等待/唤醒机制。

### 2. 三级物理存储结构

锁表通过三层核心数据结构在内存中建立“进程”与“资源”的映射：

#### A. LOCK (锁对象)

* **标识**：由 `LOCKTAG` 唯一确定（例如：库OID + 表OID）。
* **内容**：记录当前对象上**所有进程**持有的锁模式计数（如 3 个读锁，0 个写锁）以及等待队列。
* **索引**：通过全局哈希表 `LockMethodLockHash` 查找。

#### B. PROCLOCK (授权对)

* **标识**：由 `(LOCK, PGPROC)` 二元组确定。
* **内容**：记录**特定进程**在**特定对象**上持有的具体锁模式。
* **作用**：它是 `LOCK` 和 `PGPROC` 的交汇点，用于快速定位某个进程到底拿了哪几把锁。

#### C. PGPROC (进程结构)

* **内容**：每个进程专属的结构，包含该进程持有的 `PROCLOCK` 链表。
* **作用**：当事务提交或回滚时，系统通过此链表快速释放该事务持有的所有重量级锁。

---

### 3. 访问流程：哈希分区与 LWLock

为了降低高并发下的锁竞争，锁表被划分为多个**分区（Partitions）**（通常为 16 个）：

1. **Hash 计算**：对 `LOCKTAG` 进行哈希，确定其所属分区。
2. **物理保护**：申请该分区对应的 `partition LWLock`（轻量级锁）。
3. **数据操作**：在受保护的分区内进行 `LOCK` 或 `PROCLOCK` 的增删改。
4. **释放**：操作完成立即释放 `LWLock`。

---

### 4. 关键字段对照表

| 结构层级 | 存储字段示例 | 核心用途 |
| --- | --- | --- |
| **LOCKTAG** | `relid`, `dbid` | 定义“锁的是什么” |
| **LOCK** | `grantMask`, `waitCursor` | 定义“当前被谁锁了”和“谁在排队” |
| **PROCLOCK** | `holdMask`, `procLink` | 定义“这个进程具体拿了什么模式” |
| **PGPROC** | `myProc->myXactLocks` | 事务结束时，按图索骥“一键释放” |

## 快速路径 (Fast-Path) 优化

### 1. 设计初衷与背景

* **痛点**：在高并发只读场景（如大量 `SELECT`）下，所有进程频繁申请 `AccessShareLock`，会导致锁表的分区轻量级锁（Partition LWLock）成为瓶颈。
* **核心思想**：**“自己的锁，记在自己家里”**。对于无冲突的弱锁，跳过全局哈希表的操作，直接登记在进程私有的内存数组中。

### 2. 存储位置与结构

Fast-Path 锁不再存储在全局 `LOCK` 或 `PROCLOCK` 中，而是保存在每个进程的 **`PGPROC`** 结构体中：

* **`fpRelId[16]`**：一个固定长度为 16 的数组，存储所锁定表的 OID。
* **`fpLockBits`**：一个 64 位的位图（Bitmask），记录每个 OID 对应的锁模式（每个槽位占 4 位，记录 1-4 级弱锁模式）。
* **`fpLocalTransactionId`**：记录当前虚拟事务 ID，用于标识该 Fast-Path 锁的所有者。

---

### 3. 工作流程：申请与校验

#### A. 快速申请 (Fast-Path Acquisition)

1. **模式判定**：检查锁模式是否为弱锁（Mode < 4，如 `AccessShare`）。
2. **槽位查找**：在 `fpRelId[]` 中寻找空位。
3. **本地登记**：将表 OID 存入数组，并在 `fpLockBits` 中标记模式。
4. **完成**：无需申请分区 LWLock，无需访问共享内存哈希表。

#### B. 强锁冲突处理 (Conflict Handling)

当某个进程需要对表加 **强锁**（如 `AccessExclusiveLock`）时：

1. **扫描全局**：强锁进程必须遍历 `ProcArray` 中的所有活跃进程。
2. **检查 Fast-Path**：查看每个进程的 `fpRelId[]` 是否包含目标表。
3. **强制迁移 (Cleansing)**：如果发现有 Fast-Path 持有者，强锁进程会将其“踢出”私有数组，并转换（Transfer）为全局锁表中的 `PROCLOCK`。
4. **禁止后续 FP**：在该强锁释放前，该对象上的 Fast-Path 路径被暂时封禁。

---

### 4. 优化效果对比

| 维度 | 重量级锁表路径 | 快速路径 (Fast-Path) |
| --- | --- | --- |
| **同步原语** | 分区 LWLock (原子竞争) | 无或极简原子操作 |
| **内存访问** | 全局哈希表查找 (多次读写) | 本地数组操作 (单次写) |
| **冲突开销** | 每次加锁都要检测冲突 | 仅在有人申请强锁时才扫描 |
| **典型性能** | 约数百纳秒至微秒 | 数十纳秒 |

---

### 5. 局限性

* **容量限制**：每个进程最多只能通过 Fast-Path 持有 16 个不同表的锁。超过 16 个后，自动退化为重量级锁表路径。
* **对象限制**：仅适用于 `Relation`（表）类型的锁，不适用于 `TransactionId` 或 `VXID` 锁。
